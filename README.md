# Context Dependency in LLM-Based Requirements Clarification

This repository contains the replication package for an exploratory study on
context dependency in LLM-based requirements clarification within a
Specification-Driven Development (SDD) workflow.

The study investigates how different types of contextual information affect the
clarification behavior of an LLM-based SDD agent, operationalized with Spec Kit
and the `/speckit.clarify` command.

## Study Overview

The experiment analyzes four User Stories under six context conditions. The
goal is to understand whether contextual information reduces necessary
clarification needs, produces redundant questions, or reveals new emergent
gaps.

| Dimension | Value |
|---|---:|
| User Stories | 4 |
| Context conditions | 6 |
| Executions | 24 |
| Raw clarification questions | 177 |
| Average questions per execution | 7.38 |
| Minimum / maximum questions per execution | 3 / 10 |
| Primary questions after duplicate handling | 118 |
| Normalized informational gaps | 42 |
| Judges | 3 |

## Research Questions

- **RQ1.** How does contextual information affect the number of necessary and
  unanswered clarification gaps produced by an SDD agent?
- **RQ2.** To what extent do lexical, operational, decisional, and systemic
  contexts eliminate gaps associated with their respective informational
  functions?
- **RQ3.** What patterns of persistence, redundancy, and emergence are observed
  after contextual information is introduced?

## Context Conditions

Each User Story was executed under six conditions:

| Code | Context supplied to the agent |
|---|---|
| `C0` | User Story only, without additional context |
| `CL` | Lexical context: domain terms and expressions |
| `CO` | Operational context: rules, limits, and exceptions |
| `CD` | Decisional context: prior decisions, rationales, and rejected alternatives |
| `CS` | Systemic context: actors, dependencies, interfaces, and related requirements |
| `CT` | Literal union of CL + CO + CD + CS |

```

with files such as:

```

## Annotation and Adjudication

The unit of analysis is the **normalized informational gap**, not the literal
clarification question.

A single question may contain more than one informational need, and different
questions may refer to the same underlying gap. Therefore, the extracted
questions were normalized and grouped using `Gap_ID`.

Each gap was classified along three dimensions:

| Dimension | Values |
|---|---|
| Contextual category | lexical, operational, decisional, systemic, unassociated |
| Informational coverage | unanswered, partially answered, redundant |
| Functional relevance | necessary, useful, non-pertinent |

The extracted questions were classified by three judges. Final labels were
assigned by unanimity when all judges agreed and by majority vote when two
judges agreed. Cases without majority or involving coverage/relevance
uncertainty were manually reviewed.

Duplicate questions were preserved for traceability but excluded from metric
computation. The 118 primary questions correspond to rows in
`collected-data/questions_adjudicated.csv`; the 42 normalized gaps correspond
to `collected-data/gaps_normalized.csv`.

## Behavior Across Conditions

After normalization, gaps observed in the baseline condition `C0` were compared
with the contextual conditions `CL`, `CO`, `CD`, `CS`, and `CT`.

| Behavior | Definition |
|---|---|
| `Eliminated` | Gap appeared in C0 and disappeared after context was introduced |
| `Persistent` | Gap appeared in C0 and continued even though an answer was available or partially available |
| `Emergent` | Gap did not appear in C0 but appeared after context was introduced |
| `Unchanged` | Gap appeared in both conditions, but the added context was not intended to resolve it |

## Main Preliminary Finding

The combined-context condition (`CT`) produced the lowest number of necessary
and unanswered gaps, followed by the operational context condition (`CO`).
However, `CT` also produced the highest number of emergent gaps, suggesting that
richer context may reduce unresolved clarification needs while simultaneously
revealing new dependencies, exceptions, or constraints.

## Known Limitations

- The study is exploratory.
- Only four User Stories were analyzed.
- Each User Story-condition pair was executed once.
- Only one SDD workflow and one underlying model were used.
- LLM outputs may vary across re-executions.
- The analysis is descriptive and does not report inferential statistics.
- The normalized gap labels and adjudicated classifications are interpretive and
  depend on human judgment.
